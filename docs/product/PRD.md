## 2. 产品概述

AlphaQuest 是一个面向 Web3 项目方和币圈用户的增长任务平台。

平台一端服务 B 端 Web3 项目方，为其提供项目宣发、任务增长、链上交互激励、用户获取、数据分析、API/SDK 快速接入、链上公平奖励池等能力。

平台另一端服务 C 端用户，用户可以通过签到、邀请、完成项目任务、链上交互、学习答题、内容创作等方式获得平台积分，并使用积分参与项目方赞助的权益活动、奖励池、白名单活动或平台权益兑换。

### 2.1 一句话定位

> AlphaQuest 是 Web3 项目方的增长基础设施，也是币圈用户发现项目、完成任务、赚取积分、参与链上公平奖励的任务平台。

### 2.2 核心价值主张

对项目方：

1. 快速创建增长任务活动。
2. 低成本获取真实 Web3 用户。
3. 验证用户链上行为和社交行为。
4. 通过 API/SDK 快速接入任务和积分能力。
5. 使用链上公平奖励池提升活动可信度。
6. 获得可复盘的数据分析和用户画像。

对 C 端用户：

1. 每天发现新的 Web3 项目和任务机会。
2. 通过真实贡献获得积分。
3. 使用积分参与项目方奖励活动。
4. 通过邀请、签到、学习、链上交互持续积累权益。
5. 通过链上可验证开奖机制获得公平感和信任感。

对平台：

1. 沉淀 Web3 用户流量池。
2. 形成项目方增长服务收入。
3. 形成积分、任务、风控、开奖、API 的基础设施能力。
4. 长期沉淀用户声誉数据和项目分发网络。

## 3. 项目背景

### 3.1 项目方痛点

1. 冷启动难，缺少早期用户。
2. 社区增长依赖 KOL 和空投，成本高且效果不稳定。
3. 获得的用户质量低，刷子和女巫账号多。
4. 任务活动搭建成本高，需要自己开发系统。
5. 缺少一套可复用的任务、积分、邀请、开奖、数据工具。
6. 任务完成结果难以验证，尤其是链上行为与社交行为。
7. 用户数据分散在 X、Discord、Telegram、链上浏览器和表格中，难以统一分析。
8. 抽奖或奖励活动缺少可信机制，用户容易质疑黑箱。

### 3.2 C 端用户痛点

1. 想发现早期项目、空投机会、测试网任务、白名单活动。
2. 想通过时间和贡献换取积分、奖励、权益和潜在机会。
3. 不信任中心化抽奖平台。
4. 害怕被虚假项目、钓鱼项目、低质量项目欺骗。
5. 希望有一个聚合任务的平台，每天都可以打开完成任务。
6. 希望自己的链上行为和社区贡献能形成长期声誉。

### 3.3 平台机会

1. 面向亚洲和中文 Web3 市场做更本地化的增长平台。
2. 提供比普通任务平台更强的 API/SDK 接入能力。
3. 用链上公平奖励池建立信任差异化。
4. 用风控和用户声誉系统提升项目方投放效率。
5. 把平台从任务站升级为项目增长基础设施。

## 4. 产品目标

### 4.1 MVP 阶段目标

MVP 阶段需要验证以下核心假设：

1. 项目方愿意为任务增长、用户获取和活动工具付费。
2. C 端用户愿意每天完成任务获取积分。
3. 积分可以有效提升用户留存和任务完成率。
4. 邀请机制可以带来低成本增长。
5. 链上公平奖励池可以增强用户信任并提升活动参与度。
6. API/SDK 能让项目方更快接入平台能力。

### 4.2 MVP 成功指标

| 指标 | 目标 |
|---|---:|
| 首批合作项目 | 10 - 30 个 |

| 钱包连接用户 | 20,000+ |
| 月活用户 | 15,000+ |
| 日活用户 | 3,000+ |
| 月任务完成量 | 200,000+ |
| 单活动平均参与人数 | 2,000+ |
| 首批 B 端收入 | 30,000 - 100,000 USDT |
| 付费项目方复购率 | 30%+ |
| 作弊账号拦截率 | 60%+ |
| 活动开奖争议率 | 低于 1% |

## 6. 用户角色

### 6.1 C 端用户

#### 新手用户

特征：刚进入 Web3 或任务平台，对钱包和链上操作不熟悉。

需求：快速上手、低门槛拿积分、清楚知道每个任务怎么完成、避免复杂链上交互。

#### 空投猎人

特征：熟悉测试网、链上交互和项目任务，对积分、白名单、潜在空投敏感。

需求：高奖励任务、新项目机会、链上任务聚合、更高等级权益、活动公平透明。

#### 内容创作者

特征：擅长写推文、教程、视频、Meme、项目分析。

需求：内容任务、创作奖励、被项目方发现、长期声誉积累。

#### 社区推广用户

特征：有一定社群资源，擅长邀请和传播。

需求：邀请链接、邀请海报、邀请数据、明确的邀请奖励。

### 6.2 B 端项目方

#### 新项目团队

需求：快速冷启动、低成本获取第一批用户、增加社群关注、完成测试网交互、收集钱包地址和反馈。

#### 成熟项目团队

需求：精细化用户增长、高质量链上用户、API 接入、数据分析、活动复盘、白标任务中心。

#### 公链 / 生态方

需求：推动生态项目增长、鼓励链上交互、做生态任务季、分发生态激励、形成生态用户画像。

### 6.3 平台运营人员

需求：审核项目方、审核任务、管理活动、处理风控、配置推荐位、查看平台数据、处理用户申诉。

## 7. 核心业务流程

### 7.1 C 端用户主流程

```text
访问平台
  → 连接钱包
  → 创建账户
  → 绑定社交账号，可选
  → 完成新手任务
  → 获得积分
  → 浏览项目任务
  → 完成任务
  → 积分入账
  → 邀请好友
  → 参与奖励池
  → 等待开奖
  → 查看链上开奖结果
  → 领取奖励
  → 提升等级和声誉
```

### 7.2 项目方主流程

```text
注册项目方账号
  → 提交项目资料
  → 平台审核
  → 创建 Campaign
  → 配置任务
  → 配置积分奖励
  → 配置奖池，可选
  → 发布活动
  → 用户参与
  → 查看数据
  → 导出有效用户
  → 活动复盘
  → 复购或升级套餐
```

### 7.3 链上公平奖励池流程

```text
项目方创建奖励池
  → 项目方托管奖品到合约
  → 用户消耗或锁定积分获得 tickets
  → 后端生成 ticket 区间
  → 分批提交 batch root，可选
  → 活动达到目标积分
  → 提交 final merkle root
  → 请求 VRF 风格随机数
  → 计算 winning ticket
  → 公布中奖结果
  → 中奖用户提交 proof claim 奖励
```

### 7.4 邀请奖励流程

```text
用户 A 生成邀请链接
  → 用户 B 通过链接注册
  → B 绑定钱包
  → B 完成有效任务
  → B 通过基础风控
  → A 的邀请奖励进入 pending
  → 活动结束或风控确认
  → A 的邀请奖励释放
```

## 9. 用户端需求

### 9.1 登录与账户

#### 钱包登录

支持钱包：MetaMask、WalletConnect、OKX Wallet、Binance Web3 Wallet、TokenPocket 可选。

需求说明：

1. 用户点击连接钱包。
2. 前端发起签名请求。
3. 后端验证签名。
4. 若钱包首次登录，则自动创建用户。
5. 若钱包已存在，则登录现有账户。
6. 登录后生成 access token 和 refresh token。

验收标准：

1. 用户可使用钱包完成注册和登录。
2. 同一钱包不会重复创建多个账户。
3. 签名内容包含 nonce、域名、时间戳，防止重放攻击。
4. 用户可以断开钱包。

#### 用户资料字段

| 字段 | 是否必填 | 说明 |
|---|---|---|
| wallet | 是 | 用户主钱包地址 |
| username | 否 | 用户昵称 |
| avatar | 否 | 头像 |
| email | 否 | 邮箱 |
| twitter | 否 | X 账号 |
| telegram | 否 | Telegram 账号 |
| discord | 否 | Discord 账号 |
| referral_code | 是 | 系统生成邀请码 |
| user_level | 是 | 用户等级 |
| risk_score | 是 | 风控分 |

### 9.2 首页

首页需要让用户快速看到：今日可做任务、热门项目、高奖励任务、即将开奖活动、积分余额、邀请入口。

模块：

1. 顶部导航。
2. 用户状态卡片。
3. 每日签到卡片。
4. 热门任务区域。
5. 即将开奖区域。
6. 新项目区域。
7. 邀请卡片。
8. 排行榜，P1。

### 9.3 任务中心

任务分类：每日任务、新手任务、社交任务、链上任务、学习答题、内容创作、邀请任务、高奖励任务、项目深度体验。

任务状态：available、locked、in_progress、pending_review、completed、failed、expired、rewarded。

任务详情页展示：任务标题、所属项目、任务说明、完成步骤、奖励积分、任务类型、截止时间、参与人数、审核方式、风险提示、操作按钮。

不同任务验证方式：

| 任务类型 | 验证方式 |
|---|---|
| 签到 | 系统自动验证 |
| 关注 X | 第三方 API 或用户授权验证 |
| 加入 Telegram | Bot 验证 |
| 加入 Discord | OAuth / Bot 角色验证 |
| 链上交易 | 交易哈希 + 事件监听 |
| 持有 NFT | 链上余额查询 |
| 答题任务 | 系统判分 |
| 内容任务 | AI 初审 + 人工复审 |
| 反馈任务 | 人工审核 |
| 邀请任务 | 关系链 + 风控验证 |

### 9.4 签到系统

用户每天可以签到一次，获得基础积分。

签到奖励建议：

| 连续天数 | 奖励 |
|---|---:|
| 第 1 天 | 1 AP |
| 第 2 天 | 2 AP |
| 第 3 天 | 3 AP |
| 第 4 天 | 4 AP |
| 第 5 天 | 5 AP |
| 第 6 天 | 6 AP |

防作弊规则：同一用户每天只能签到一次；同一设备多账号签到进入风控；同一 IP 大量签到进入风控；高风险用户签到积分进入 pending。

## 10. 积分系统

### 10.1 积分名称

暂定名称：Alpha Points，简称 AP。

### 10.2 积分性质

对外规则：

1. AP 是平台服务积分。
2. AP 不等于稳定币。
3. AP 不可提现。
4. AP 不可转让。
5. AP 不可交易。
6. AP 不承诺升值。
7. AP 仅用于平台任务、活动、权益、奖励池参与等场景。

### 10.3 积分类型

| 类型 | 说明 | MVP 是否支持 |
|---|---|---|
| Earned Points | 完成任务获得 | 是 |
| Promo Points | 平台活动赠送 | 是 |
| Locked Points | 冻结积分 | 是 |
| Paid Credits | 用户购买积分 | 否，第二阶段再考虑 |

### 10.4 积分状态

| 状态 | 说明 |
|---|---|
| pending | 待确认 |
| available | 可用 |
| locked | 已锁定 |
| spent | 已消耗 |
| reversed | 已冲正 |
| expired | 已过期 |

### 10.5 积分流水要求

每一笔积分变动必须生成流水。

字段：transaction_id、user_id、amount、direction、point_type、status、source_type、source_id、balance_after、risk_score、created_at、confirmed_at。

核心规则：

1. 积分发放必须有来源。
2. 积分扣除必须有用途。
3. 所有积分变动不可物理删除，只能冲正。
4. 高风险积分进入冻结状态。
5. 邀请积分延迟释放。
6. 活动取消后积分自动退回。

## 11. 邀请系统

### 11.1 功能说明

用户可以通过邀请链接邀请好友注册平台，并在好友完成有效行为后获得邀请奖励。

邀请链接示例：

```text
https://alphaquest.io/invite/ABC123
```

### 11.2 绑定规则

1. 用户首次注册时，如果 URL 包含邀请码，则绑定邀请关系。
2. 一个用户只能绑定一个邀请人。
3. 已注册用户不能补绑邀请人。
4. 同设备、同 IP、同钱包簇的邀请关系进入风控。
5. 平台管理员可人工解除异常邀请关系。

### 11.3 邀请奖励类型

| 奖励类型 | 触发条件 | 状态 |
|---|---|---|
| 注册奖励 | 被邀请人注册并绑定钱包 | pending |
| 有效任务奖励 | 被邀请人完成指定数量任务 | pending / available |
| 活动参与奖励 | 被邀请人参与活动并消耗积分 | pending |
| 活动成功奖励 | 活动开奖且被邀请人无作弊 | available |

### 11.4 MVP 建议规则

1. 被邀请人完成注册：邀请人获得 1 AP，pending。
2. 被邀请人完成 3 个有效任务：邀请人获得 5 AP。
3. 被邀请人参与奖励池并活动开奖成功：邀请人获得被邀请人有效参与积分的 3% - 5%。
4. 每个邀请人每日邀请奖励上限：100 AP。
5. 每个被邀请人只产生一次基础邀请奖励。
6. 高风险邀请奖励延迟 72 小时释放。

## 12. 奖励池系统

### 12.1 功能定位

奖励池不是平台主业务本身，而是任务平台中的链上公平奖励模块。

对外建议命名：Fair Draw、Lucky Launch、Alpha Reward Pool、链上公平奖励池。

不建议在产品主文案中过度使用“夺宝”“充值夺宝”“赢 USDT”等高风险表达。

### 12.2 活动类型

| 类型 | 说明 | MVP 是否支持 |
|---|---|---|
| 积分抽签池 | 用户用任务积分参与，随机开奖 | 是 |
| 白名单池 | 随机抽取白名单资格 | 是 |
| NFT 奖励池 | 抽取 NFT 奖励 | 是 |
| USDT 奖励池 | 项目方托管 USDT 奖励 | 是，需规则审查 |
| 固定兑换 | 用户用积分兑换固定权益 | P1 |
| 免费参与池 | 用户通过免费入口参与 | 是 |

### 12.3 活动状态

| 状态 | 说明 |
|---|---|
| draft | 草稿 |
| pending_review | 待平台审核 |
| scheduled | 已排期 |
| open | 进行中 |
| filled | 达到目标积分 |
| sealed | 名单已封存 |
| random_requested | 已请求随机数 |
| finalized | 已开奖 |
| claiming | 可领取 |
| completed | 已完成 |
| cancelled | 已取消 |
| disputed | 争议处理中 |

### 12.4 参与规则

1. 用户必须登录并连接钱包。
2. 用户必须满足活动参与条件。
3. 用户使用 available AP 参与。
4. 每 1 AP 可获得 1 个 ticket，具体比例可由活动配置。
5. 项目方可设置单用户参与上限。
6. 用户参与后，积分进入 locked / spent 状态。
7. 活动取消时，积分退回。
8. 活动开奖后，积分不退回。
9. 高风险用户可被限制参与。
10. 合规要求下必须支持免费参与入口。

### 12.5 公平开奖规则

1. 达到目标积分后，活动停止接收新参与。
2. 系统生成最终参与名单和 ticket 区间。
3. 系统生成 Merkle Root。
4. 合约记录 Merkle Root 和 totalTickets。
5. 合约请求 VRF 风格随机数。
6. 随机数返回后计算 winningTicket。
7. 根据 ticket 区间确定中奖用户。
8. 中奖用户可提交 proof 领取奖励。
9. 结果页面展示完整验证信息。

开奖计算：

```text
winningTicket = randomNumber % totalTickets + 1
```

请求头：

```http
X-AQ-Key: aq_xxx
X-AQ-Timestamp: 1714380000

```

签名字符串：

```text
METHOD + PATH + BODY_HASH + TIMESTAMP + NONCE
```

```http
POST /v1/campaigns
POST /v1/tasks
GET /v1/users/{wallet}/eligibility
POST /v1/tasks/{task_id}/verify
POST /v1/points/grant
GET /v1/users/{wallet}/points

```

### 14.5 Webhook

## 15. 链上合约需求

### 15.1 合约目标

链上合约只负责关键可信环节：奖池托管、活动规则承诺、Merkle Root 记录、随机数请求和验证、开奖结果确定、奖励领取。

高频任务、积分和用户行为仍在链下处理。

### 15.2 合约模块

#### PrizeVault

职责：奖池托管。

功能：depositPrize、lockPrize、releasePrize、refundPrize、emergencyPause。

#### DrawRoundManager

职责：活动状态管理和开奖。

功能：createRound、commitBatch、sealRound、requestRandomness、fulfillRandomness、finalizeRound、claimPrize。

#### MerkleVerifier

职责：验证用户 ticket 区间。

验证字段：roundId、user、startTicket、endTicket、proof。

#### AccessControl

角色：DEFAULT_ADMIN_ROLE、OPERATOR_ROLE、PAUSER_ROLE、ROUND_CREATOR_ROLE、TREASURY_ROLE。

### 15.3 合约状态

```solidity
enum RoundStatus {
    Created,
    Open,
    Filled,
    Sealed,
    RandomRequested,
    Finalized,
    Claiming,
    Completed,
    Cancelled
}
```

### 15.4 Claim 验证

用户领取奖励时，合约检查：

1. round 已 finalized。
2. 用户提交的 proof 有效。
3. winningTicket 在用户 ticket 区间内。
4. 用户未领取过。
5. 奖池余额充足。

## 16. 风控系统需求

### 16.1 风控目标

1. 降低刷子和女巫攻击。
2. 提高项目方用户质量。
3. 避免邀请系统被批量薅羊毛。
4. 保护奖池公平性。
5. 降低平台合规和舆情风险。

### 16.2 风控评分维度

| 维度 | 说明 |
|---|---|
| 钱包年龄 | 钱包创建和首次交易时间 |
| 链上活跃度 | 交易次数、交互合约数量 |
| 资产情况 | 是否为空钱包、是否批量钱包 |
| 设备指纹 | 是否多账号同设备 |
| IP 风险 | 代理、机房、批量注册 |
| 社交账号质量 | 账号年龄、粉丝、活跃度 |
| 任务行为 | 完成速度、重复模式 |
| 邀请图谱 | 异常裂变结构 |
| 内容相似度 | 是否复制粘贴 |
| 历史违规 | 是否被封禁或冲正 |

### 16.3 风控等级

| 等级 | 分数 | 动作 |
|---|---:|---|
| 低风险 | 0 - 30 | 正常 |
| 中风险 | 31 - 60 | 积分 pending / 延迟释放 |
| 高风险 | 61 - 80 | 积分冻结 / 限制参与奖励池 |
| 极高风险 | 81 - 100 | 禁止参与 / 封禁 / 人工审核 |

### 16.4 风控动作

延迟积分发放、冻结积分、拒绝任务通过、限制邀请奖励、禁止参与奖励池、封禁账号、生成风控工单。

## 19. 非功能需求

### 19.1 性能需求

1. 首页加载时间小于 2 秒。
2. 任务列表接口响应小于 500ms。
3. 积分流水查询小于 1 秒。
4. 活动高峰支持 10,000 QPS，可逐步扩展。
5. 单活动支持 100 万级参与记录。
6. API 可用性目标 99.9%。

### 19.2 安全需求

1. 钱包签名登录防重放。
2. API HMAC 签名。
3. 重要操作二次确认。
4. 管理后台 RBAC 权限。
5. 敏感数据加密。
6. 积分流水不可物理删除。
7. 合约管理员权限使用多签。
8. 合约支持暂停。
9. 合约上线前安全审计。
10. 日志审计可追溯。

### 19.3 合规需求

1. 积分不可提现。
2. 积分不可交易。
3. 积分不可转让。
4. 不承诺收益。
5. 奖励池需要公开活动规则。
6. 高风险地区限制参与。
7. 必要时提供免费参与入口。
8. 涉及推广任务需披露赞助关系。
9. 用户数据导出需要权限控制。
10. 项目方必须通过基础审核。

## 29. 核心产品原则

1. 不做充值夺宝平台，做 Web3 增长任务平台。
2. B 端项目方付费是主要商业模式。
3. C 端用户积分是留存和激励工具，不是金融资产。
4. 链上只处理公平性和资产托管，不承载所有高频业务。
5. 邀请系统只做一级奖励，不做多级返佣。
6. 风控系统从第一天就要建设。
7. 项目方 API/SDK 是长期规模化关键。
8. 原生 App 不是 MVP 必需品。
9. 所有奖励活动必须规则清晰、可验证、可审计。
10. 平台长期护城河是项目方资源、用户流量、风控、数据和 API 生态。

## 30. 当前版本结论

AlphaQuest MVP 应优先完成以下闭环：

```text
项目方创建活动
  → 配置任务
  → 用户完成任务
  → 获得积分
  → 邀请好友
  → 参与项目方奖励池
  → 链上公平开奖
  → 用户领取奖励
  → 项目方查看数据
  → 项目方复购
```

只要这个闭环成立，平台就具备继续扩展到 API、Widget、Telegram Mini App、白标 SaaS、声誉系统和大型生态任务季的基础。

MVP 不应追求功能大而全，而应优先验证：

1. 项目方是否愿意付费。
2. 用户是否愿意持续完成任务。
3. 积分是否能形成留存。
4. 链上公平开奖是否提升信任。
5. 风控是否能压制刷子。
6. 数据看板是否能支撑项目方复购。
